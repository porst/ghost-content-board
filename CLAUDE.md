# 跟 Claude Code 協作開發指南

> 這份指南整理自鬼磕頭內容管理系統的實際開發過程（PR #1-#9），
> 記錄下真正踩過的坑，讓下次開發更順、更省成本。
>
> **使用方式**：把這份文件放在專案 Repository 的根目錄，命名為 `CLAUDE.md`，
> Claude Code 開始工作時會自動讀取並照著執行，不需要每次重新提醒。

---

## 零、溝通原則：怎麼下指令最省成本

這幾條是跟 Claude 協作時的通用習慣，不限於這個專案：

1. **不確定時先問，不要用猜的直接動工**：如果需求不清楚（例如「這個要串LINE還是網頁按鈕？」），應該先問 1-3 個關鍵問題，而不是選一個方向硬做，做完才發現猜錯要重來。
2. **小改動用小改動的方式處理**：只是改錯字、調整文案，不需要重新規劃架構或大幅討論，直接改就好。「規劃先行」是給複雜功能用的，不是每次都套用同一套流程。
3. **累積的決策要能被回顧**：像這份文件本身，就是把過程中做過的關鍵決定（Edge Runtime要換Node.js、要不要重複利用編號等）記錄下來，之後不用重新想一次、也不用重新解釋一次。
4. **改動範圍大時，主動拆解**：一次要做的事情太多、太複雜時，應該先拆成幾個小任務、排出優先順序，而不是一次全塞進去，出錯了也不知道是哪一段壞的。
5. **不要為了流程而流程**：規劃、self-check這些習慣的目的是省成本，如果套用某個固定格式反而讓每次回應變長、變制式化，那就是本末倒置，簡單問題就該簡單回答。

---

## 一、開工前：先問架構問題，不要直接開始做

每次要新增功能，先確認三件事，讓 Claude Code 主動問清楚再動工：

1. **這個功能需要後端嗎？** 如果是純顯示的網頁（像 GitHub Pages 靜態頁面），任何需要「寫入資料」或「呼叫外部 API」的功能都需要後端支援，不是加個按鈕就好。
2. **這個頁面是公開的嗎？** 如果是公開網址、沒有登入機制，任何會花錢或會改資料的功能，都需要加一層簡單的密碼保護，不然任何人都能亂用。
3. **執行環境有沒有限制？** 這次踩過兩次真實的坑：
   - Vercel Edge Runtime 不支援動態程式碼生成（`new Function`）
   - `@anthropic-ai/sdk` 用到 Node 專屬模組，不能跑在 Edge Runtime，要改用 Node.js Runtime
   - **教訓**：用到官方 SDK 或比較重的套件時，先確認它跟部署環境相容，不要等部署失敗才發現。

---

## 二、除錯時：一次把資訊給完整，減少來回

**這是這次消耗最多 token / usage 的地方。**

除錯時盡量一次提供：
- 完整的錯誤訊息（不要摘要，直接複製貼上）
- 是在哪個操作觸發的（例如「我點了 XX 按鈕」「我在 LINE 群組發了 XX 訊息」）
- Log 截圖或文字（Vercel Logs、GitHub Actions Logs 都可以直接複製貼上）

**避免**：先講「壞掉了」再一輪一輪補資訊，這樣 Claude Code 要來回猜測、重新讀取 repo，會多花好幾倍的資源。

---

## 三、部署前的檢查清單

每次要合併 PR、部署新功能，照這個順序做：

1. **確認所有需要的環境變數都設定好了**（API Key、Token、密碼等），漏設通常是「功能悄悄失敗但不報錯」的最大原因
2. **合併 PR 到 main**
3. **確認 Production 部署狀態是綠色 Ready，不是紅色 Error**（Vercel Deployments 頁面）
4. **實際操作測試一次**，不要只看「部署成功」就當作完成——部署成功只代表程式碼沒有語法錯誤，不代表功能邏輯是對的
5. 如果是 API 相關功能，**去 Logs 頁面確認真的有請求進來、有沒有錯誤**，不要只憑感覺判斷

---

## 四、安全性：公開頁面的基本防護

如果專案是「公開靜態頁面 + 背後 API」這種架構（像這次的 GitHub Pages + Vercel Edge Function）：

- 任何會**寫入資料**或**花費 API 額度**的端點，一定要加密碼驗證（例如 `BOARD_EDIT_TOKEN` 這種簡易 token）
- 使用者輸入的內容（例如 LINE 訊息）顯示在網頁上時，記得做 HTML escape，避免被惡意內容攻擊（XSS）
- GitHub Token、API Key 這類敏感資訊只能放在 Vercel/GitHub 的環境變數，絕對不要寫進程式碼裡

## 五、費用觀念：分清楚兩種花費

- **Claude.ai / Claude Code 用量**：走訂閱方案（Pro/Max），除錯、寫程式、來回測試都算在這裡，這是這次消耗最大的部分
- **API 費用**（`ANTHROPIC_API_KEY`）：走量計費，通常是排程自動化、或功能本身呼叫 API（如「重新生成內容」）才會用到，正常使用下每月大概幾十元台幣等級，不用過度擔心

**省錢的關鍵不是少用功能，而是減少「除錯的來回次數」**——這才是這次真正燒掉最多資源的地方。

---

## 六、給非工程師背景使用者的提醒

- 遇到 Claude Code 回報「已經測試過、mock 過」，這通常代表**邏輯測試過，但沒有連到真實服務測試過**（因為沙盒環境常常連不到外部網路）。這種情況下，合併後**一定要自己實際操作測試一次**，不要假設一定成功。
- 遇到部署失敗，先複製完整的錯誤訊息給 Claude Code，不要自己先猜測原因或動手改設定——很多時候看起來像 A 問題的錯誤，其實是完全不同的 B 問題（例如這次的 404 錯誤訊息指向 `line-webhook`，但其實問題出在完全沒被提到的 `regenerate-script`）。
- 「合併 PR」跟「重新整理網頁確認生效」是兩個分開的動作，合併完不代表你馬上就能在瀏覽器看到最新結果，有時候需要等 1-2 分鐘部署完成。

## 附錄：這次專案實際踩過的坑（案例記錄）

| 問題 | 根本原因 | 解法 |
|------|---------|------|
| Vercel Function 一直卡 25 秒逾時 | 對外請求沒有設定 timeout，程式卡死不報錯 | 每個 fetch 都加 timeout + AbortController |
| `new Function` 解析資料失敗 | Edge Runtime 禁止動態程式碼生成 | 改用 `JSON.parse` |
| `@anthropic-ai/sdk` 部署失敗 | SDK 用到 `node:fs`/`node:path`，Edge Runtime 不支援 | 該支 API 改用 Node.js Runtime |
| LINE Webhook 沒反應 | Webhook URL 被清空、或 Use webhook 開關關閉 | 確認 LINE Developers Console 設定 |
| Bot 自動退出群組 | 「Allow bot to join group chats」設定關閉 | 開啟該設定後重新邀請 |
| Claude API timeout | SDK 預設 timeout 太短（10-20秒），不夠複雜生成任務用 | 延長 SDK timeout + Vercel maxDuration |
